            *   **排序**：在任何“合同剩余天数”筛选条件下，已到期客户都应优先于“90天内”的客户进行展示。
            *   **操作**：提供更明确的处理路径，如“紧急续签”或“转未续签反馈”。同时，“已到期”应作为客户状态或标签在列表中明确标识。

*   **【P2】在服员工构成有“汇总少于明细”的潜在问题 (`PRD` `纪要`)**

    *   **建议**: 以 `纪要` 结论为准。`PRD` 中的“确保求和相等”是一个低优先级且难以达成的目标。
        1.  **更新 `PRD`**：将“确保全部在服 = 各构成数量之和”修改为“在服构成各Tab页的数量之和可能大于或小于‘全部在服’数量，这是由于业务数据多状态共存导致的，客服需知晓”。

#### 3. 技术与交互层面的潜在问题

*   **【P1】资金信息脱敏实现方案复杂 (`PRD` `纪要`)**
    *   **问题**: `PRD` 中资金信息的脱敏方案是“点击后需判断是否内网访问，仅内网支持...”。`纪要` 讨论了更复杂的方案（通过内网域名请求，外网访问直接404）。
    *   **风险**: 这种“点击一个按钮，判断网络环境，再发起新请求”的交互模式，存在明显的延迟和技术风险。
        1.  **用户体验差**：点击后无明显反馈，用户不知道请求是否成功发起。
        2.  **外网状态不明确**：外网环境下，用户点击按钮后，是直接“无反应”，还是看到“toast提示”（PRD方案），或看到404页面（纪要讨论方案）？前端开发需要非常明确的交互逻辑。
        3.  **开发复杂度高**：需要对前后端服务进行复杂的网络环境感知设计。
    *   **建议**: 采用更简洁、可靠的方案。
        *   **最终方案**：建议全部以“¥****”或“¥***万”形式展示。前端不再提供“查看明文”的交互按钮。若需要查看，可提供一个“前往 [原系统名] 查看详情”的链接。这样既保证了安全性，又明确了“不可查看”的预期。
        *   **若坚持内网查看方案**：必须明确以下交互细节：
            1.  **反馈机制**：点击后，立即加载loading。若网络环境不符，toast提示“请在公司内网下查看”。若请求成功，替换脱敏值为明文。
            2.  **网络判断**：前端如何判断？是发起一个专门的、仅内网可达的探针请求？还是后端接口返回的数据格式不同（外网返回‘-’，内网返回真实值），前端根据值做判断？建议后端接口统一返回，由前端判断。

*   **【P2】列表排序与筛选的“跨源”限制未在原型和前端逻辑中体现 (`PRD` `纪要`)**
    *   **问题**: `PRD` 明确指出“由于跨系统，HC数、在服人数、风险事项无法进行筛选和排序”。但 `原型(index.html)` 中，客户视图的列表字段包含“在服人数...在招HC...风险事项”等无法排序的字段，且原型未明确提示用户无法排序。
    *   **建议**: 前端交互要符合实际能力边界。
        1.  **排序提示**：在客户视图列表的“在服人数”、“在招 HC”等表头旁，不应提供可点击的排序箭头（如原型所示）。
        2.  **隐藏排序逻辑**：或在表格底部用极简文字提示：“列表排序规则：优先按风险数量，其次按在服人数...”。让用户清晰地知道排序机制，而不是看到无法点击的排序箭头。

*   **【P2】员工视图-工单进展“+N”的交互体验模糊 (`PRD`)**
    *   **问题**: `PRD` 9.1.5 提到“多工单显示 +N...点击后，打开右推弹窗，展示所有主工单卡片及操作按钮”。
    *   **风险**: 如果员工有10个工单，都挤在一个“右推弹窗”里展示“工单卡片”，是否可行？弹窗是否会过长？操作按钮如何呈现？
    *   **建议**: 将“右推弹窗”改为一个“半屏页(Modal Sheet)”或“抽屉页(Drawer)”，而不是一个小的弹窗。
        1.  **布局**：使用侧边抽屉，显示完整的工单列表。
        2.  **展示**：每个工单卡片建议只展示核心信息（工单类型/名称、当前节点、创建时间），并提供一个“查看详情”链接跳转到原系统。
        3.  **操作**：高频操作（如“确认进度”、“驳回”等）可以作为按钮直接放在卡片上，并遵循工单类型对应的权限控制。

##### 模块 2：工单处理页 (`工单`)

-   **功能点 ID**: `WRK-01`
-   **角色**: 客服
-   **核心逻辑**:
    -   复用SSC工作台的工单一级分类和筛选样式。
    -   数据范围：仅展示派给**当前用户**的**待执行**子工单。
-   **排序规则**:
    1.  ZOC（Offer审核）已逾期子工单置顶。
    2.  按工单更新时间**升序**（越早越靠前）。
    3.  工单编号升序（兜底）。
-   **操作权限**: 用户在工作台可见/可操作的按钮，必须与其在原iStaffing系统中该工单的按钮权限保持一致。

##### 模块 3：客户服务跟进 (`客户`)

-   **功能点 ID**: `CUS-01` (客户视图)
-   **核心逻辑**:
    -   展示用户有权限的所有客编列表。
    -   **排序规则**：优先按`风险数量`降序，其次按`在服人数`降序，最后按客编升序。
    -   **限制**：`在招HC`、`在服人数`、`风险数量`等从其他系统获取的字段，**不支持排序和筛选**。
-   **交互**:
    -   点击“在服人数” → 跳转至员工视图并自动填充客编。
    -   点击“风险事项”数量 → 跳转至风险视图并按风险类型筛选。
    -   操作按钮：“门户预览”、“服务订单”、“Paas”。

##### 模块 4：员工用工跟进 (`员工`)

-   **功能点 ID**: `EMP-01` (员工视图)
-   **核心逻辑**:
    -   展示“待入职”和“在服员工”两个Tab页，默认选中“在服员工”。
    -   “在服员工”Tab页下，支持按“全部在服”、“在职”、“试用期”、“待离职”、“续签中”、“转移中”进行拆分。

-   **排序规则**: 优先按风险数量降序 > 按入职日期升序 > 按试用剩余时间升序 > 按合同剩余时间升序 > 按用工ID升序。

    -   特殊标签（ER/三期/工伤）。
    -   用工信息：客户、职位、地点、入职/离职日期。
    -   工单进展：展示**主工单**（最多1条）；多条显示+N，点击可展开。
-   **功能点 ID**: `EMP-02` (风险视图 - 合同即将到期)

    -   无续签工单的员工显示“发起续签”按钮；有则显示“查看续签进展”按钮。

    -   点击员工可展开二级列表，展示该员工各欠款账单月的明细（账单年月、编号、应税工资等）。

##### 模块 5：权限与系统集成 (`权限`)

-   **功能点 ID**: `AUTH-01`
-   **角色**: 客服、Leader
-   **核心规则**:
    -   **数据范围**: 客服 → fastgo个人客编权限。Leader → fastgo汇报线下所有下级用户的客编权限。
    -   **快捷入口权限**: 在工作台自有数据库中维护菜单配置，通过fastgo提供的接口( token + menu code )实时判断用户在目标系统是否有权限。
    -   **操作按钮权限**: 工单操作按钮沿用iStaffing的按钮权限判断。其他操作按钮不做独立控制，由原系统拦截无权限请求。

#### 三、 关键业务流程梳理

1.  **员工欠款计算流程 (复杂，需多团队对齐)**
    *   **数据源**: 结算系统收方账单（已过账），账单月从2025年12月开始。
    *   **计算单位**: 员工 + 账单月。
    *   **月度金额公式**: `应税工资 + 年终奖合计 + 税后增加 + 非居民数月奖金 - 工资调整已使用金额`。
    *   **欠款区间判断**:
        *   逐月累加。当某月金额首次 < 0，触发欠款区间。
        *   从触发月开始，持续累加后续月份的金额，直至累计值 >= 0，则该欠款区间结清。
        *   存在未结清区间的员工，计为欠款员工。无论多少欠款月，一个员工仅计为一个欠款单位。
    *   **刷新频率**: 每日定时刷新。

2.  **员工合同续签流程**
    *   **场景**: 客服在“员工风险视图→合同即将到期”页面查看。
    *   **决策**:
        *   若员工无进行中的续签工单 → 点击 `发起续签` → 跳转到iStaffing续签页面（自动填充员工信息）。
        *   若员工有进行中的续签工单 → 展示工单进展 → 点击 `查看续签进展` → 跳转至iStaffing续签工单详情。
    *   **注意**: 该流程由客服人工判断后操作，工作台本身不自动发起任何流程。

**前置依赖与风险提示：**

1.  **外部接口交付风险**：大量任务（如BE-01至BE-10, BE-16）依赖外部团队（fastgo、招聘、员管、结算、数仓、工单系统）的接口交付。项目管理者需确保这些接口在8月初完成评审并于8月初交付，否则8月底的里程碑将无法达成。
2.  **汇报线性能风险**：Leader角色的数据范围（BE-03）依赖fastgo汇报线接口，该接口**当前性能差**，需技术委员会推动解决。若无法解决，可考虑先将Leader功能降级（如仅展示其个人数据），或增加缓存与异步刷新机制。
3.  **员工欠款复杂性**：BC-10的计算逻辑复杂，涉及多张表和历史数据。建议后端团队与结算团队（超哥、凤超、戴朋）专门对齐一次。测试时需准备足够多的边界数据（如结清、未结清、刚结清、跨月欠款等）。
4.  **敏感数据处理**：风险金余额的脱敏方案（内网/外网判断）是前后端联调的高风险点。建议统一方案：后端接口始终返回明文或掩码，前端不判断网络环境。若为安全必须内网，建议采用内网专有域名访问，而非前端做复杂判断。
