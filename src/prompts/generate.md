# 任务：生成排期方案

请根据以下文档生成完整的项目排期方案。

# 分析维度

1. 任务拆解 — 将功能点拆解为可独立开发、独立验收的开发任务
2. 工时评估 — 基于量化基准逐项估算，并说明估算依据
3. 依赖分析 — 识别任务间的前后依赖关系和可并行项
4. 测试策略 — 为每个任务设计验收用例，为复杂计算提供标准测试数据
5. 风险识别 — 标注未决策事项对工时的影响

# 量化标准

## 精确功能拆分的定义

- 每个任务只对应**一个可交付物**：一个页面、一个组件、一个 API 接口、一个数据模型
- 任务内部不包含可独立测试的子功能（如果一个任务包含多个可测试环节，必须拆开）
- 前后端职责分离：同一个功能的前端页面、后端接口、联调对接必须是独立任务
- 非功能性工作单独拆出：权限体系、埋点、性能优化、通用组件、异常处理各自独立

## 合理排期时间的定义

工时公式：工时 = 基础开发时间 + 自测(×20%) + 代码评审(×10%)

基准参考（中级开发 3年经验）：

### 前端页面

前端工时 = 表单/逻辑复杂度基准 + 设计稿还原复杂度基准，两者取较大值

#### A. 表单/逻辑复杂度

| 分类 | 判定条件（满足全部） | 基准工时 |
|------|------------------|--------|
| 基础表单页 | 字段 ≤8 个，全部标准控件，列表仅分页+1个筛选 | 0.5~1 人天 |
| 标准 CRUD 页 | 字段 ≤15，含文件上传或富文本，2~4个筛选+批量操作 | 1~2 人天 |
| 复杂逻辑页 | 满足任一：字段 >15、动态表单、树形选择、内联编辑、复杂状态机 | 2~3 人天 |

#### B. 设计稿还原复杂度（最耗时的部分，必须单独评估）

| 分类 | 判定条件 | 额外工时 |
|------|----------|--------|
| 无设计稿 | 根据 PRD 自行设计 UI，使用现有 UI 库默认风格 | +0 人天 |
| 轻量设计稿 | 有设计稿但布局简单（单栏/双栏），无自定义动效，无响应式要求 | +0.5~1 人天 |
| 中等设计稿 | 多栏布局、自定义组件样式、1~2 处动效、响应式适配 | +1~2 人天 |
| 重度设计稿 | 满足任一：复杂动效/动画、全屏滚动、可视化图表定制、多端适配(PC+H5+小程序)、自定义主题系统 | +2~4 人天 |

#### C. 技术栈系数调整

根据技术栈实际情况调整基准工时：
- 使用成熟 UI 库（Ant Design/Element Plus）且设计稿贴合组件库：×1.0
- 使用 UI 库但设计稿大量自定义样式：×1.3~1.5
- 自研 UI 或无组件库：×1.5~2.0
- H5 移动页 / 小程序（相对 PC Web）：×1.2~1.5
- 桌面应用（Electron/Tauri）：×1.1~1.3（含原生 API 集成）

#### D. 前端专家视角——拆分前置检查

在拆分前端任务前，必须先从前端专家角度审视：
1. **组件复用分析**：哪些组件可以复用？复用是否已有现成组件，还是需要新开发？
2. **状态管理复杂度**：页面间是否共享状态？是否需要全局状态管理？
3. **设计稿还原工作量**：设计稿中有多少自定义样式/动效？与 UI 库的差距有多大？
4. **平台适配工作量**：是否需要响应式？多端适配？
5. **技术风险**：是否有未用过的第三方库、未实现过的交互模式？
6. **设计稿参照分析**：如果提供了 HTML 设计稿，必须参照设计稿中的实际布局和组件来评估工时，而非仅根据 PRD 文字描述。设计稿中的组件数量、自定义样式、布局复杂度是工时的核心依据。

### 后端接口

| 分类 | 判定条件（满足全部） | 基准工时 |
|------|------------------|--------|
| 单表 CRUD | 仅操作 1 张表，WHERE 条件 ≤3 个，无事务、无外部调用 | 0.5~1 人天 |
| 多表关联接口 | JOIN 2~3 张表，或有 1 次外部服务调用，或有简单事务 | 1~2 人天 |
| 复杂业务接口 | 满足任一：JOIN >3 张表、含复杂计算/状态机/定时任务、并发控制、多级缓存 | 2~3 人天 |

### 联调

| 分类 | 判定条件 | 基准工时 |
|------|----------|--------|
| 单接口对接 | 1 个前端页面调 1 个后端接口，数据结构直接映射 | 0.5 人天 |
| 流程联调 | 满足任一：多接口串联、状态同步、错误重试、WebSocket/轮询 | 1~2 人天 |

### 非功能性

| 分类 | 基准工时 |
|------|--------|
| 权限体系（角色模型+权限矩阵+中间件） | 2~3 人天 |
| 通用组件封装（设计+实现+文档+单测） | 1~2 人天 |

## 估算纪律

- **不要刻意压缩**：如果一个页面包含 5 个交互模块，不能合并为 1 个任务只给 1 人天
- **不要刻意放大**：简单的 CRUD 不需要给 3 人天
- **必须说明依据**：每个任务的工时旁边，用一句话说明为什么是这个时间（参考基准的哪一行）

# 输出模板

## 一、功能点级任务明细

这是最细粒度的拆分，每个功能点独立估时。先按前端/后端/联调分组，每组内按页面/模块组织。

### 拆分规则
- 每个任务必须独立可运行、具备原子性
- 如果一个任务预估超过 3 人天，必须拆为子任务
- 纯前端能独立完成的工作（UI/交互/样式/本地状态）必须标为 🟢[前端]，不要混入联调
- **一个功能点可能拆出多个任务**：如“订单列表页”可拆为“列表页 UI 开发”+“筛选组件开发”+“批量操作开发”

字段说明：
- **所属组件/页面**：该任务归属的上层组件或页面名称（用于后续汇总）
- 复杂度：S(≤4h) / M(4~16h) / L(16~24h) / XL(>24h需拆分)
- 预估工时(小时)：精确到小时，区间值取中位数
- 预估工时(人天)：按 8h=1人天 换算，保留1位小数
- **估算依据**：参考上方基准表的哪一行，以及为什么偏上/偏下（1句话）。如有设计稿，在此处注明设计稿中该页面/组件的实际情况（如“设计稿含8列表格+2筛选+弹窗表单”）
- 前置依赖：依赖的任务ID，或“外部：XX团队接口”
- 验收标准：怎样算“做完了”（1句话）

# 质量要求

- 每个任务必须独立可运行、独立验收，具备原子性（不可再拆分）
- 纯 UI/交互/样式工作必须标为 🟢[前端]，不要把前端能独立完成的工作标为联调
- 每个任务必须有明确的验收标准
- 每个任务必须有“估算依据”字段，说明参考了哪个基准
- 同一功能的前端、后端、联调必须拆为独立任务，不能合并
- 禁止出现“开发XX功能 5人天”这样的粗粒度任务，必须拆开为具体子任务
