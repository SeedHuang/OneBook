# HTML 设计稿分析增强设计

## 背景

当前 OneBook 在需求分析和工时估算时，支持导入 HTML 文件作为文档。但存在两个问题：

1. **readHtml 暴力去标签**：`file.service.ts` 的 `readHtml()` 函数删除所有 HTML 标签，只保留纯文本。UI 设计稿中的布局、组件、交互信息全部丢失。
2. **提示词无设计稿分析维度**：4 个提示词 md（system/review/extract/generate）的分析维度全部面向文本型 PRD，没有指导 AI 如何分析 HTML 设计稿。

结果：HTML 设计稿在功能点分析和工时估算中被完全忽略。

## 目标

让 AI 在需求分析时能：
- 识别 HTML 设计稿的页面布局、UI 组件和交互元素
- 将设计稿与 PRD 文档交叉验证，发现遗漏和不一致
- 参照设计稿的实际复杂度评估前端还原工时

## 设计

### Task 1：readHtml 改造——结构化页面描述

**文件**：`electron/services/file.service.ts`

**当前行为**：正则删除所有 HTML 标签，返回纯文本。

**目标行为**：解析 HTML DOM 结构，输出一份 Markdown 格式的结构化页面描述文档，包含以下部分：

```
# HTML 设计稿分析

## 页面结构
- 整体布局方式（单栏/双栏/三栏/栅格）
- 各区域功能和尺寸关系
- 响应式设计情况

## UI 组件清单
- 逐个列出页面中的 UI 组件
- 标注组件类型（表格/表单/按钮/弹窗/菜单/标签页等）
- 标注组件的关键属性（列数、字段数、选项数等）

## 可见文本内容
- 页面标题、表头、按钮文字、标签文字等
- 去重去空后的结构化文本列表

## 交互元素
- 可点击元素及其预期行为
- 表单提交、弹窗触发、Tab 切换等交互流程

## 设计复杂度评估
- 自定义样式程度（轻量/中等/重度）
- 动效/动画情况
- 预估还原难度
```

**技术方案**：

- 使用 `cheerio` 库解析 HTML DOM（轻量级，无需浏览器环境）
- 提取逻辑：
  - 布局容器：识别含 `display:flex`/`display:grid`/class 含 layout 关键词的元素
  - 语义化标签：`<table>`、`<form>`、`<button>`、`<select>`、`<input>`、`<nav>`、`<header>`、`<aside>` 等
  - Ant Design 组件类名：识别 `.ant-table`、`.ant-form`、`.ant-modal`、`.ant-menu` 等 class 前缀
  - 可见文本：遍历叶节点提取 `textContent`，过滤空白和重复内容
  - 交互属性：`onclick`、`href`、`data-action`、`type="submit"` 等
- 保留 `<script>` 和 `<style>` 的删除逻辑
- 保留 HTML 注释的删除逻辑
- 新增依赖：`cheerio`

### Task 2：提示词补充——设计稿分析维度

**文件**：`src/prompts/system.md`

在"阶段二：逐文档深度分析"末尾增加：

```markdown
## 设计稿分析（当文档类型为 HTML 设计稿时）

- **页面布局识别**：整体布局方式（单栏/双栏/三栏/栅格）、各区域功能和尺寸关系
- **UI 组件清点**：逐个列出页面中的 UI 组件（表格、表单、按钮、弹窗、菜单、标签页等），标注组件类型和数量
- **交互元素提取**：识别所有可交互元素（点击、悬停、展开、切换），梳理交互流程
- **与 PRD 交叉验证**：将设计稿中的功能模块与 PRD 文档的功能点逐一对应，标注设计稿中有但 PRD 未提及的功能，以及 PRD 中提到但设计稿未体现的功能
- **前端还原复杂度预判**：基于组件数量、自定义样式程度、动效需求，给出初步还原难度评级（轻量/中等/重度）
```

**文件**：`src/prompts/extract.md`

在功能点列表的输出模板表格中增加两列：

```markdown
| 功能点ID | 所属模块 | 功能名称 | 实现端 | 关联角色 | 核心业务规则 | 数据来源 | 交互说明 | 设计稿参照 | 设计稿还原难度 |
```

- 设计稿参照：`✅ 设计稿有对应UI` / `❌ 设计稿未覆盖`
- 设计稿还原难度：轻量 / 中等 / 重度 / `-`（无设计稿时）

**文件**：`src/prompts/generate.md`

在"前端专家视角——拆分前置检查"中增加第 6 项：

```markdown
6. **设计稿参照分析**：如果提供了 HTML 设计稿，必须参照设计稿中的实际布局和组件来评估工时，
   而非仅根据 PRD 文字描述。设计稿中的组件数量、自定义样式、布局复杂度是工时的核心依据。
```

在工时表格中增加"设计稿参照"字段：

```markdown
| 任务ID | 所属组件/页面 | 任务名称 | 实现端 | 关联功能点 | 复杂度 | 预估工时(小时) | 预估工时(人天) | 设计稿参照 | 估算依据 | 前置依赖 | 工作内容描述 | 验收标准 |
```

### Task 3：文档类型标识传递

**文件**：`src/components/ChatPanel.tsx`

**当前行为**：文档上下文组装格式为 `### ${d.name}\n${d.content}`，AI 无法区分文档类型。

**目标行为**：根据文档的 `type` 字段添加类型前缀标签：

```
### 📄 需求文档：项目PRD.md
文档内容...

---

### 🎨 设计稿：workspace.html
（以下为 HTML 设计稿的结构化描述）
文档内容...

---

### 📊 数据文档：数据字典.xlsx
文档内容...
```

**映射规则**：
- `md` → `📄 需求文档`
- `html` → `🎨 设计稿`
- `xlsx` → `📊 数据文档`

**依赖**：`documentStore` 中的文档对象已有 `type` 字段（由 `file.service.ts` 的 `readFileContent` 返回），需确认该字段是否在存储时保留。如未保留，需在文档导入和存储流程中补充 `type` 字段的持久化。

### Task 4：文档类型持久化（条件任务）

需检查 `documentStore.ts` 和 `db.service.ts` 中文档存储逻辑：
- 如果 `documents` 表没有 `type` 字段，需新增该列
- 如果 IPC 传递文档时未携带 `type`，需在 handlers 中补充

## 不变的部分

- `readMarkdown()`、`readExcel()` 保持不变
- `review.md` 暂不修改（需求审查的核心维度已覆盖文档间一致性检查，设计稿分析主要在 extract 和 generate 阶段发挥作用）
- 快捷分析按钮逻辑不变
- 对话/消息持久化逻辑不变

## 风险

- HTML 设计稿体积较大时，结构化描述也会较长，可能消耗较多 token。需在实际使用中观察，必要时对描述进行截断或摘要。
- cheerio 解析非标准 HTML 时可能出错，需确保 mockup HTML 格式规范。
